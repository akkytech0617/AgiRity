# Linear タスク作成手順 (SOP)

## 1. 目的
- 設計ドキュメントからタスクを抽出し、Linearプロジェクトに登録する方法を標準化する。
- 各Issueに明確な作業計画を含め、手戻りを防ぐ。

## 2. 前提条件
- **Team**: [Team Name] (Key: [TEAM_KEY])
- **Project**: AgiRity

## 3. Issue作成フロー

### Step 1: タスク抽出
設計書 (RFC/ADR/Requirements) から実装タスクをリストアップする。
- **粒度**: 1タスク = 0.5日〜3日程度で完了するサイズ
- **分割**: 実装、テスト、ドキュメントは分けず、1つの機能単位にするのが基本

### Step 2: 重複チェック
Linear上でキーワード検索し、似たタスクがないか確認する。

### Step 3: Issue作成 (Template)
以下のフォーマットでDescriptionを記述する。

```markdown
## 概要
[何を、なぜやるのか簡潔に]

## 受け入れ基準
- [ ] [基準1]
- [ ] [基準2]

## 作業計画
1. [Step 1]
2. [Step 2]

## テスト戦略
- [ ] Unit Test: `src/xxx.test.ts`
- [ ] E2E Test: (必要な場合)

## 参考資料
- [RFC-XXX](link)
- [Figma](link)
```

### Step 4: 優先度設定
| Priority | 意味 |
|---|---|
| **Urgent (1)** | 今すぐ対応が必要（バグ、ブロッキング） |
| **High (2)** | 次のマイルストーンに必須 |
| **Medium (3)** | 通常の機能開発 |
| **Low (4)** | Nice-to-have、将来的な改善 |

## 4. ツール活用

### CLIからの作成 (推奨)
AgiRityプロジェクトでは、Linear CLIやAIエージェント経由での作成を推奨する。

```bash
# チームIDの取得 (初回のみ)
# ブラウザの開発者ツール等で確認が必要
```

## 5. ベストプラクティス
- **タイトル**: `[Feature] ワークスペース作成機能` のようにプレフィックスをつけると分かりやすい。
- **ステータス**: 作成時は `Backlog` または `Triage`。
- **担当者**: 基本的に着手する人がアサインする（作成時は未定でも可）。
